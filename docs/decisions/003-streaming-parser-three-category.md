# ADR 003: 流式解析器按三类标记分别处理

## 日期

2026-06-03

## 状态

已采纳（架构待实现）

## 背景

`StreamingBlockParser` 当前的实现策略是"先写段落，再在 `\n` 时纠正"：

```
逐字符输入 → 默认写到段落（乐观） → \n 时用完整行判断块类型 → 从段落里"抠"出内容，替换/转换块
```

这个策略导致了三个问题：

1. **`OnLineEnd` 职责过重**：同时做代码块围栏闭合、ATX 标题检测+段落拆分、列表/分割线/Setext 转换、块引用行处理。一个函数混了 5-6 种互不相关的逻辑。

2. **标题流式空帧**：因为标题在 `\n` 才被"发现"，而 `\n` 不产生内容字符，`NotifyContentChanged` 刷新时无新内容。上一个修复（空格处提前创建标题）绕过了 `OnLineEnd`，说明 `\n` 的时机对于某些块类型来说太晚了。

3. **`_lineBlockKind` 语义模糊**：`null` 同时表示"未确定"和"已确定但尚未创建块"，`Feed` 循环中的条件分支链复杂。

同时，在分析流式体验时发现 Markdown 标记天然可分为三类，各自适用不同的处理策略。

## 决策

将 Markdown 标记按其特征分为三类，分别由独立的处理路径负责：

### 第一类：行首式（Line-Start Markers）

**定义**：出现在行首、决定整行/整块结构类型的标记。

**包含**：`#`（ATX 标题）、`>`（块引用）、`- ` / `* ` / `+ `（无序列表项）、`---` / `===`（分割线/Setext）、` ``` ` / `~~~`（围栏代码块）、行首缩进。

**处理策略**：前移至字符流中做即时检测。

```
新行开始 → 行首检测阶段
  ├─ 检测到明确行首标记（如 #+空格 / > / -+空格 / ```）
  │    → 立即创建对应块类型（非段落），后续字符直接流式写入
  └─ 经过 N 个字符仍未匹配任何行首模式
       → 回落为段落，后续字符流式写入
```

**关键原则**：不创建段落作为"临时容器"。要么直接确定具体块类型，要么回落为段落——没有"先写进去再刨出来"的过程。

**延迟权衡**：行首字符在检测完成前不会出现在 UI 中。但在真实 AI 流式场景中，HTTP SSE 下发的 token 块通常包含 2-10 个字符，行首检测几乎在到达时即完成，延迟不可感知。测试所用的 180ms/字符是刻意放大的场景，不影响实际体验。

### 第二类：包围式（Surrounding Markers）

**定义**：包裹内联内容的格式标记。

**包含**：`*...*` / `_..._`（强调）、`**...**` / `__...__`（粗体）、`` `...` ``（代码范围）、`[text](url)`（链接）、`![alt](url)`（图片）。

**处理策略**：保持现状。`InlineParser` 已是独立类，`InlineCollection` 通过延迟解析在渲染时调用。行解析器只需透传原始内容，不感知包围式标记。

### 第三类：换行符（Line Breaks）

**定义**：行结束/段落分隔的结构信号。

**包含**：`\n`、空行（`\n\n`）、两空格+`\n`（硬换行）。

**处理策略**：缩减为纯状态转换器。

```
\n → 当前块定稿（或标记可延续）
    → 重置行首状态
    → 等待下一行首个字符
```

当行首式标记的检测前移到字符流后，`\n` 不再需要做"发现块类型"的工作。其职责从"发现+转换"简化为"定稿+重置"。

## 架构示意

```
Feed(char)
  │
  ├─ 行首状态（_lineStart）
  │    ├─ 缓冲行首字符 → 匹配模式 → 创建对应块
  │    └─ 不匹配 → 回落到段落
  │
  ├─ 流式写入（块类型已确定 → StreamCharToBlock）
  │    └─ 内容透传到 InlineCollection（包围式由 InlineParser 处理）
  │
  └─ \n（OnLineEnd）
       └─ 块定稿 + 状态重置（不做类型检测/转换）
```

## 影响

正面：
- `OnLineEnd` 的复杂度显著降低，不再需要做段落拆分/类型转换
- 每种块类型在字符流中即时创建，内容直接从正确类型的块中流式输出
- 空帧问题从架构层面消除（块类型在 `\n` 之前已确定）
- 新增行首块类型时，只需在行首检测模块中加一条模式匹配，不影响其他路径

负面：
- 行首字符在检测完成前有短暂延迟（真实场景不可感知）
- `-` 和 `*` 仍有歧义（`- item` vs `---`），少数边界仍需 `\n` 辅助判断

## 备选方案

- **备选 A（当前方案）**：保持"先写段落再在 `\n` 纠正"。缺点是 `OnLineEnd` 持续膨胀，每个新增块类型都需要在 `OnLineEnd` 和 `TryConvertParagraphAtLineEnd` 中加分支。
- **备选 B（完全缓冲行首）**：每行开头在 `\n` 之前不输出任何字符。对非特殊行延迟过大（整行缓冲完才显示），体验差。

## 后续步骤

1. 将行首检测逻辑从 `OnLineEnd` / `TryConvertParagraphAtLineEnd` 中提取为独立的 `LineStartDetector` 模块
2. `OnLineEnd` 精简：移除 ATX 检测、段落转换等逻辑，专注于定稿+重置
3. 处理 `-` / `*` 的歧义边界（分割线 vs 列表项）作为行首检测模块的内部逻辑
4. 测试覆盖：验证真实 SSE 块大小下发时的流式正确性
